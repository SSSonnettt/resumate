## Architecture

**Builder For Web** is a WYSIWYG page builder for resumes. Built with **Vue 3**, **Vuetify 3**, **TypeScript**, and **Vite 5**.

### Two sheet modes (core concept)

The builder operates in one of two modes, set via `workshopStore.snapshot.mode` (`SHEET_MODE` enum in `src/constant/enum.ts`):

- **RESUME** (`"resume"`): Flow layout, fixed 820px width. Components stack vertically. Uses `vue-draggable-plus` for reordering. Components are in `src/pages/drop-components/resume/`.
- **ABSOLUTE** (`"absolute"`): Free-positioning mode with 8-point resize handles and grid-line background. Components use CSS `position: absolute` with pixel coordinates. Driven from `src/pages/compoonents/AbsoluteSheet.vue`.

A third mode FLOW exists in the enum but is currently hidden in the UI (`Toolbar.vue` mode selector).

### Main page layout (3-panel)

`src/pages/index.vue` splits the builder into three panels:

1. **DragSheet** (`src/pages/compoonents/DragSheet.vue`): Left sidebar listing draggable components. Reads available components from `src/pages/drop-components/index.ts`, which dynamically glob-imports `.vue` files grouped by mode directory.

2. **DropSheet** (`src/pages/compoonents/DropSheet.vue`): The canvas where components are placed. Delegates to either `ResumeSheet` or `AbsoluteSheet` based on mode.

3. **AttrSheet** (`src/pages/compoonents/AttrSheet.vue`): Right sidebar property editor for the selected component. Renders `CustomForm` fields generated by the component's CSS attribute class.

### Component attribute system

Each drop-component carries an `attrs.style` object that is an instance of a CSS attribute class defined in `src/pages/attrDefined.ts`. Class hierarchy:

```
BaseAttr (basic CSS: width, height, color, border, etc.)
  ├── GlobalAttr (adds zoom, aspectRatio)
  ├── BaseFlowAttr (adds display, flex alignment, text alignment)
  │     ├── DefaultFlowtAttr
  │     └── ImgFlowAttr
  └── BasePositionAttr extends BaseFlowAttr (adds top/left, zIndex)
        ├── DefaultPositiontAttr
        └── ImgPositionAttr
```

Each class exposes `getFormFields()` returning an array of `FormField` descriptors that drive the property editor UI. Store hydration in `workshop.ts` reconstructs these class instances from JSON by matching `className`.

### State management

- **Pinia** store (`src/stores/`):
  - `workshop.ts`: Core snapshot state — the entire builder document (`SnapshotVariable`). Includes `dropComponents` array, global settings (theme, gridLine), undo/redo indexes, and UI state (`activeIndex`).
  - `app.ts`: App-level theme preference, persisted to localStorage.

- **Undo/redo**: Implemented via Immer's `produceWithPatches` (`src/hooks/immer.ts`). Each mutation to `workshopStore.snapshot` is wrapped in `setSheetState()`, which captures patches and inverse patches for undo/redo. State is auto-saved to `localStorage` under key `WORKSHOP_LOCALSTORAGE_KEY`.

### Routing

File-based routing via `unplugin-vue-router` — pages in `src/pages/` become routes automatically:
- `/` → `src/pages/index.vue`
- `/preview` → `src/pages/preview/index.vue`

Layouts are applied by `vite-plugin-vue-layouts`. The default layout is `src/layouts/default.vue`.

The preview page (`/preview`) exposes `window.loadConfig(config)` for external integration — an external consumer can call this to replace the snapshot at runtime, then poll `window.loadConfigState === "updated"` to know when rendering is complete.

### API layer

Uses **Alova** (similar to Axios) configured in `src/api/alova.js`:
- Base URL: `/rest` (proxied to `localhost:3000`)
- Manual content-type negotiation
- PDF download trigger: auto-detects `application/pdf` response and triggers browser download
- `src/api/index.js`: Currently only exposes `getPdf(config)` — POSTs snapshot JSON to `/rest/pdf`

### Type system

Shared types are in `types/workshop.d.ts` (aliased as `@/workshop` via tsconfig paths):
- `DropCopmonent`: A placed component instance (id, code, label, attrs, events, config, transitions, animations)
- `SnapshotVariable`: The entire builder document
- `WorkshopVariable`: The Pinia store state
- `FormField`: Descriptor for rendering a property editor field

### Drop component registration pattern

Each `.vue` file under `src/pages/drop-components/` exports a `componentInfo` object that registers it as a draggable component:

```ts
export const componentInfo = {
  code: "Header",           // unique component code (matches filename)
  label: "简历头部",         // display name
  icon: "mdi-page-layout-header",  // MDI icon
  path: SHEET_MODE.RESUME,  // which mode directory this belongs to
  type: SHEET_MODE.RESUME,  // ABSOLUTE components have type="absolute"
  attrs: {
    style: new DefaultFlowtAttr({ width: "100%" }),  // initial CSS attr instance
  },
  config: { name: "XXX", ... },  // component-specific data (e.g., text content)
}
```

`src/pages/drop-components/index.ts` glob-imports all `componentInfo` exports, grouping them by parent directory name (mode). The `DragSheet` reads from this grouped object via `componentInfos[workshopStore.snapshot.mode]`.

When dropped onto the canvas, the `componentInfo` object is cloned (via Immer's `createDraft`) with a new `id` and pushed into `snapshot.dropComponents`.

**Resume mode components** (under `resume/`) receive these props at render time: `id`, `events`, `config`, `isPreview`. They are rendered directly via dynamic `<component :is="...">` in `ResumeSheet.vue`.

**Absolute mode components** are loaded asynchronously via `useAsyncDropComponent()` (`src/hooks/asyncComponent.ts`), which uses `defineAsyncComponent` with loading/error states and calls `setSpotStyle()` after each component mounts.

### Property editing undo flow

`AttrSheet.vue` uses a **focus/blur snapshot pattern** for undo support:

1. `formFocus(code)` is called on field focus — captures the current value in a local `temp` variable
2. User edits the field (v-model updates the store directly for real-time preview)
3. `formBlur(code)` is called on field blur — restores the original value, then calls `setSheetState()` which re-applies the new value through Immer, producing a proper undo patch

This avoids creating an undo entry for every keystroke while still showing live preview.

`CustomForm.vue` is a dynamic form renderer that maps `FormField.type` to the appropriate Vuetify form control (select, checkbox, radio, range-slider, textarea, color, number, text).

### SCSS conventions

- Global SCSS variables injected via `vite.config.mts` → `additionalData: @import '@/styles/variables.scss'`
- Vuetify theme settings in `src/styles/settings.scss`
- CSS custom properties for theme colors defined in `App.vue` (`--theme-color`, `--theme-bg-color`)
